## Context

项目目标是交付 `$greeting-generator` Skill 包，发布目录为 `greeting-generator/`。现有文档给出了完整业务流程和输入输出约束，但缺少可执行的实现分层与落地任务。实现需要兼顾两类输入通道：对话内输入与文件路径输入；同时保证输出结构固定、内容自然且候选之间有差异。

## Goals / Non-Goals

**Goals:**

- 形成稳定的多阶段交互流程，严格覆盖节日信息、用户画像、用途规格、发送对象信息采集。
- 支持批量发送对象输入（Markdown/JSON/YAML/txt 等可解析文本），并提供字段缺失的可控回退策略。
- 生成按发送对象分组的 Markdown 结果，满足每位对象 `N` 条候选、总数为 `对象数 x N`。
- 提供可选写入文件能力，默认文件名包含 skill 名称与时间戳。

**Non-Goals:**

- 不实现外部消息通道（短信网关、邮件网关、社交平台 API）发送能力。
- 不构建长期存储或用户画像数据库。
- 不在本阶段引入复杂模型调参或风格微调训练。

## Decisions

### 1. 采用“流程编排 + 能力函数”结构

- 决策：将 Skill 主流程拆为四段：`collect_festival`、`collect_user_profile`、`collect_usage_spec`、`collect_recipients`，随后进入 `generate_and_render` 与 `offer_export`。
- 原因：流程边界与需求步骤一一对应，便于校验缺失信息、插入默认值、编写测试。
- 备选方案：单段式大 Prompt。
- 不选原因：可维护性差，难以定位失败步骤，也不利于批量对象处理。

### 2. 为发送对象输入建立统一规范化模型

- 决策：无论来源于对话还是文件，最终归一为统一对象结构：`name/title/relation/age/gender/notes`。
- 原因：生成逻辑只消费一种内部格式，可减少分支和重复代码。
- 备选方案：按来源分两套数据结构分别处理。
- 不选原因：会导致重复校验与重复渲染逻辑，违背 DRY。

### 3. 输出模板固定，文案生成策略可变

- 决策：输出采用固定 Markdown 框架（前置上下文块 + 按对象分组列表），但每组候选通过“风格指令差异化”避免同质化。
- 原因：固定结构便于交付与持久化，策略可变保证文案不套路化。
- 备选方案：完全自由格式输出。
- 不选原因：难以校验是否满足“按对象分组、固定数量”等刚性要求。

### 4. 文件导入采用“显式支持格式 + 容错降级”

- 决策：优先解析 JSON/YAML，其次解析 Markdown/txt 的列表或段落；无法结构化时降级为“佚名”通用对象并提示风险。
- 原因：兼顾准确性与可用性，降低误解析对流程中断的影响。
- 备选方案：仅支持结构化格式。
- 不选原因：对用户输入门槛较高，降低技能可用性。

## Risks / Trade-offs

- [Risk] 非结构化文本误解析导致对象字段错位 -> Mitigation: 提供解析预览与确认环节，允许用户修正后再生成。
- [Risk] 候选差异不足导致同质化 -> Mitigation: 在生成阶段引入显式差异维度（语气、重点、句式）并进行去重检查。
- [Risk] 用户拒绝提供画像信息导致个性化不足 -> Mitigation: 明确提示“可选但影响质量”，并使用保守默认人格模板。
- [Risk] 默认文件名覆盖同名历史输出 -> Mitigation: 使用精确到秒的时间戳，并在写入前检查冲突。

## Migration Plan

1. 在 `greeting-generator/` 建立 Skill 包基础文件与目录结构。
2. 实现输入采集与对象规范化逻辑，再接入文案生成与渲染。
3. 加入文件导入、默认值与回退逻辑。
4. 增加写文件流程与默认命名规则。
5. 完成端到端示例验证，确保与规格一致后再进入后续优化。
